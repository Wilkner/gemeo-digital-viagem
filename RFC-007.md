2. Contexto e Problema
Até o momento, a interface desenvolvida (index.html) acumula funções de criação de conteúdo (caixas de texto, upload de arquivos, parametrização técnica) e de visualização. Essa sobrecarga expõe ferramentas administrativas para o visitante final e cria dependência de um banco de dados dinâmico tradicional para guardar as histórias.

Para colocar o sistema no ar de forma barata e focada em quem vai ler, precisamos:

Ocultar a área administrativa: Deixar o painel de geração de texto apenas no ambiente local (localhost).

Criar um Portal Público de Consumo: Um site focado 100% na experiência do usuário final, sem caixas de texto vazias, apenas com os seus relatos prontos e a escolha de formatos de consumo.

Eliminar custos de Banco de Dados: Armazenar as viagens prontas em arquivos estáticos de texto na pasta do próprio projeto, que serão versionados diretamente no GitHub.

3. Arquitetura Proposta (Static-File Database)
A nova estrutura de arquivos e fluxo de dados do projeto será dividida da seguinte forma:

Plaintext
PROJETO (Seu Repositório GitHub)
├── app/
│   ├── data/
│   │   └── trips/                    <── SUAS HISTÓRIAS SALVAS (.txt ou .md)
│   │       ├── david_michelangelo.txt
│   │       ├── concerto_vivaldi.txt
│   │       └── caos_milao.txt
│   │
│   ├── templates/
│   │   ├── admin.html                <── Sua interface atual (Local / Criação)
│   │   └── index.html                <── Nova interface pública (Clean / Consumo)
O Fluxo de Leitura na API:
Quando o usuário final acessar o seu site na nuvem:

O site público solicita à API quais histórias estão disponíveis.

A API lê a pasta app/data/trips/ e lista os arquivos existentes (ex: "David de Michelangelo em Florença", "Concerto de Vivaldi").

O usuário escolhe a história e o formato (Stories ou Artigo).

A API processa o arquivo estático correspondente com o Gemini e devolve a resposta formatada para a tela do visitante.

4. O Plano de Ação para Implementação
Para darmos vida a essa arquitetura, faremos o desenvolvimento em 3 etapas simples:

Passo 1 (Organização de Arquivos): Renomear seu index.html atual para admin.html (sua área privada) e criar a pasta app/data/trips/ para guardar seus relatos reais.

Passo 2 (Endpoints de Consulta Estática): Criar uma rota no backend FastAPI que lê essa pasta de arquivos e lista as viagens disponíveis para a tela de boas-vindas do usuário.

Passo 3 (O Novo Frontend Público): Criar o novo index.html focado na jornada do visitante (Boas-vindas ──► Seleção de História ──► Seleção de Formato ──► Consumo fluido).
