# RFC-004: Adaptador de Artigo Detalhado (Markdown) e Seletor de Formatos

## 1. Status
**Status:** IMPLEMENTADO
**Autor:** Wilkner
**Data:** Julho de 2026
**Área:** Camada 3 (Adaptadores de Saída) / Frontend

---

## 2. Contexto e Problema
Até a aprovação desta RFC, o Gêmeo Digital de Viagem suportava exclusivamente a geração de narrativas no formato de sequências de Stories para o Instagram (`TripStoriesOutputSchema`). 

Embora o formato de Stories seja excelente para consumo dinâmico e rápido, ele limita a capacidade do usuário de compartilhar reflexões profundas, conexões filosóficas e crônicas ricas de viagem com familiares e amigos que preferem um consumo de leitura calmo, focado e sem a urgência dos algoritmos de redes sociais. 

Além disso, a API possuía endpoints rígidos de saída (`/trips/{trip_id}/output/stories`), impedindo a expansão flexível para novos formatos sem a criação de novas rotas redundantes.

---

## 3. Objetivos
- **Multi-Formato:** Introduzir um novo adaptador de saída capaz de transformar relatos de viagem brutos em crônicas estruturadas em Markdown.
- **Unificação de Endpoints:** Consolidar as rotas de saída em um único endpoint flexível que aceita o formato desejado via parâmetro de busca (`Query Parameter`).
- **Renderização Dinâmica:** Atualizar a interface do usuário (HTML/CSS/JS) para suportar abas de visualização e renderizar Markdown nativamente na tela.

---

## 4. Arquitetura e Fluxo de Dados

A arquitetura de 3 Camadas foi mantida e expandida na Camada 3 com a introdução do `ArticleAgent`:

```text
Relato Bruto (Ingestão)
       │
       ▼
[Camada 1: Persona Service] ──► Carrega Estilo e Tom (persona_profile.json)
       │
       ▼
[Camada 3: Orquestração e Seleção de Formato (endpoints.py)]
       │
       ├─── format="stories" ──► [StoriesAgent] ──► JSON estruturado de Stories
       └─── format="article" ──► [ArticleAgent] ──► JSON estruturado de Artigo em Markdown

## 5. Especificações Técnicas

5.1. Contrato de Dados (Pydantic Schemas - app/schemas/trip.py)
Adicionou-se o schema que valida a saída do artigo estruturado e o tipo de resposta dinâmico:

class TripArticleOutputSchema(BaseModel):
    trip_id: str
    format: str = "article"
    generated_at: datetime
    title: str = Field(..., description="Um título impactante, poético e nada clichê")
    subtitle: str = Field(..., description="Subtítulo reflexivo que resume a essência da experiência")
    content_markdown: str = Field(..., description="Texto completo formatado usando Markdown estruturado")
    suggested_reading_time_minutes: int = Field(..., description="Tempo de leitura estimado em minutos")

class TripOutputResponse(BaseModel):
    output: Union[TripStoriesOutputSchema, TripArticleOutputSchema]

5.2. Design do Endpoint Unificado (app/api/v1/endpoints.py)
O endpoint /trips/{trip_id}/output/stories foi descontinuado e substituído por uma rota dinâmica:

Rota: GET /api/v1/trips/{trip_id}/output

Parâmetros de Query:

format (string, padrão: "stories"): "stories" ou "article".

mode (string, padrão: "coauthor"): "coauthor" (preserva texto original) ou "ghostwriter" (reescreve via IA).

5.3. Engenharia de Prompt do ArticleAgent (app/services/agents.py)
O ArticleAgent instrui o modelo LLM (gemini-2.5-flash) a atuar de acordo com as regras de persona e o comportamento de autoria:

Modo Co-Autor (coauthor): O LLM preserva 100% das palavras originais, atuando apenas como editor para injetar formatação de cabeçalhos (##), negritos e estimar o tempo de leitura.

Modo Ghostwriter (ghostwriter): O LLM reescreve ativamente o rascunho com base na persona do usuário.

---

## 6. Resultados e Validação técnica
Preservação de Autoria: Testado com sucesso usando o relato real "David de Michelangelo", onde o texto original foi mantido intocado, recebendo títulos inteligentes de forma editorial.

CORS e Renderização: O frontend passou a renderizar o Markdown usando a biblioteca externa marked.js direto no navegador local sem erros de requisição ou travamento de rotas.
